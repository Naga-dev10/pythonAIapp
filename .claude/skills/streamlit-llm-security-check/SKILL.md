# Streamlit + LLM API セキュリティチェック

Streamlit上で構築されたLLM API利用アプリ（Gemini, OpenAI, Claude等のSDKを使うもの）に対して、よく発生する4つのリスク領域を確認し、結果をMarkdownのセキュリティレポートとしてチャット内に出力する。

このスキルは**ユーザーが明示的にセキュリティチェックを依頼したときだけ**使う。コード変更ごとに自動的に発動させる用途ではない。

## チェックの進め方

1. アプリのエントリーポイント（`app.py`等）と `requirements.txt` / `.env` 関連ファイルを特定する。
2. 下記4観点ごとに、該当するファイル・行を実際に読んで具体的な根拠（file:line）を確認する。憶測で書かない。
3. 見つかった指摘を重大度（Critical / High / Medium / Low）で分類し、レポート形式でまとめる。
4. 該当する問題が見つからない観点も「確認済み・問題なし」として明記する（チェック漏れがないことを示すため）。

## チェック観点

### 1. APIキー・シークレットの取り扱い

- ソースコード中にAPIキーがハードコードされていないか（`grep`で `AIza`, `sk-`, `api_key\s*=\s*["']` 等のパターンを確認）
- `.env` が `.gitignore` に含まれているか。含まれていなければ重大度Highの指摘とする
- APIキーが `st.write`, `print`, ログ、エラーメッセージ等に出力されていないか（例外時にキー文字列を含むメッセージを表示してしまうケースに注意）
- サイドバーでAPIキーを入力させる箇所が `st.text_input(..., type="password")` になっているか（平テキスト表示は中程度の指摘）
- `st.secrets` を使わずに環境変数や設定ファイルを直接埋め込んでいないか（デプロイ先がStreamlit Community Cloudの場合は`st.secrets`が推奨）

### 2. プロンプトインジェクション・LLM入出力の取り扱い

- ユーザー入力がそのまま文字列結合でプロンプトに埋め込まれていないか。システム指示とユーザー入力の境界が明確か（区切り文字、システムプロンプトの分離など）
- ユーザーがアップロードしたファイルやテキストの内容を未検証でプロンプトに渡していないか
- LLMの出力を `st.markdown(..., unsafe_allow_html=True)` 等でそのままHTMLとして描画していないか（LLM出力経由のXSSリスク）
- 生成結果をそのまま `st.download_button` や外部送信に使う場合、悪意あるスクリプトや個人情報の混入チェックがあるか
- 入力長やリクエスト頻度の制限がなく、コスト超過・LLM APIへのDoS的濫用を許す構造になっていないか

### 3. 依存パッケージの脆弱性

- `requirements.txt` の各パッケージがバージョン固定されているか（無制限指定は再現性・サプライチェーンリスクの観点でMedium）
- 既知の脆弱性がないか確認する。可能であれば以下を実行して結果を読む:
  ```powershell
  pip install pip-audit
  pip-audit -r requirements.txt
  ```
  実行できない環境であれば、主要パッケージ（`streamlit`, `google-generativeai` 等）のバージョンが大きく古くないかを目視確認する
- `python-dotenv` 等、シークレット読み込みに関わるパッケージのバージョンも確認する

### 4. Streamlit特有の設定・公開範囲の問題

- デバッグ用途のコード（`st.write(os.environ)` 等、環境変数を画面に出すデバッグコード）が残っていないか
- ファイルアップロード機能がある場合、ファイルサイズ・拡張子の制限があるか
- `st.session_state` に機密情報（APIキー、ユーザー入力の個人情報等）を保持し続けていないか。特に複数ユーザーがアクセスする想定のデプロイ先では、セッション間の情報漏洩リスクに注意
- アプリを外部公開する場合、認証機構（Streamlit Community Cloudのアクセス制限、リバースプロキシでのBasic認証等）がない状態で機密性の高い機能を公開していないか
- `.streamlit/config.toml` で `server.enableXsrfProtection` や `server.enableCORS` を不必要に無効化していないか

## レポート形式

レポートはチャット内に直接Markdownで出力する（ファイルは生成しない）。以下のテンプレートに従う。

```markdown
# セキュリティチェックレポート

対象: <アプリ名 / パス>
日付: <実施日>

## 概要
<全体的な所見を2〜3文で>

## 指摘事項

### [Critical/High/Medium/Low] <タイトル>
- 場所: `file:line`
- 内容: <何が問題か>
- リスク: <悪用された場合の影響>
- 推奨対応: <具体的な修正方法>

(指摘ごとに上記を繰り返す)

## 問題なしと判断した項目
- <観点名>: <確認した内容と根拠>

## まとめ
<優先して対応すべき項目を重大度順に列挙>
```

- 重大度の目安:
  - **Critical**: 既に外部に公開されたシークレット、即座に悪用可能な脆弱性
  - **High**: シークレット管理の不備、プロンプトインジェクションで機密情報や任意操作を引き出せる構造
  - **Medium**: ベストプラクティスからの逸脱だが直接の悪用は難しいもの（バージョン未固定、平テキスト入力欄など）
  - **Low**: 望ましい改善だが緊急性は低いもの
- 推測ではなく、実際に読んだコードの行を根拠として示すこと。根拠を示せない指摘は記載しない。
